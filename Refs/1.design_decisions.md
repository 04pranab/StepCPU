# CPU Design Decisions

---

### 1 — Data width: how many bits per value?
Determines the range of numbers the CPU can hold natively. Real RISC-V is 32 or 64-bit.

Options: `1-bit` `4-bit`<- (Chosen), `8-bit`, `16-bit`, `32-bit`

Rationale: 

```4-bit is the "sweet spot"—small enough to trace by hand but large enough to run real programs (0–15). The concepts scale identically to 32-bit.```

---

### 2 — Architecture: what instruction set?
The ISA defines what instructions the CPU understands.

Options: `x86 (complex)`, `ARM (proprietary)`, `RISC-V inspired` <- (Chosen), `Custom`

Rationale: 

```RISC-V features a beautifully regular encoding, and lessons learned here transfer directly to real-world RISC-V applications.```

---

### 3 — Pipeline: single-cycle or pipelined?
Single-cycle means one instruction completes per clock tick; pipelined means multiple instructions overlap like an assembly line.

Options: `Single-cycle`<- (Chosen), `5-stage pipeline`, `Out-of-order`

Rationale: 

```Single-cycle is conceptually clean and provides a solid foundation before adding a pipeline in later steps.```

---

### 4 — Memory: Harvard or Von Neumann?
Harvard uses separate instruction and data memory; Von Neumann uses one unified memory.

Options: `Harvard`<- (Chosen), `Von Neumann`

Rationale: 

```Harvard is simpler to implement and reason about, as instructions live in ROM and data in RAM with no conflicts.```

---

### 5 — Simulation language: what order?
Python allows for fast prototyping, while SystemVerilog is used for actual hardware (RTL) design and verification.

Options: `Python first`<- (Chosen), `SystemVerilog second`<- (Chosen)

Rationale: 

```Python is for fast ideas and SystemVerilog represents hardware reality. Using both mirrors how real chip teams work.```
